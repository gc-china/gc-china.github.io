Redis 持久化

Redis 的持久化 —AOF 模式

AOF 运行原理(日志的形式)

AOF 的三种策略

  1. always (每条策略)

always——> 缓存区——> AOF文件 每条命令都会写入硬盘中

  1. everysec(每秒策略）

always——> 缓存区——> AOF文件 每秒把缓存区fsync硬盘中

  1. no

根据操作系统决定

三种策略比较

| 命令 | always | everysec | no |

| —- | ———- | ————————– | —— |

| 优点 | 不丢失数据 | 每秒一次fsync 丢失一秒数据 | 不用管 |

| 缺点 | IO开销大 | 丢一秒数据 | 不可控 |

Redis 的持久化—–Rdb 模式

  1. RDB: 在持久化本地创建一个red后缀的文件, 在恢复时,然后启动载入，等于设置了个快照

触发机制

  1. Sava(同步):这是条同步命令 在执行这条命令后 所有其他的命名必须去排队 等它执行完毕才能

- 在执行save命令后 因为他是条同步命令，其他命令必须等它执行完毕才能执行 所以当数据量过大的时候容易造成阻塞

  1. bgsave (异步) 在执行这条命令后，会返回ok 然后在后台执行

| 命令 | save | bgsave |

| ——– | ———————————- | ——————————- |

| IO 类型 | 同步 | 异步 |

| 是否阻塞 | 是 | 不是(但是阻塞发生在Fork) |

| 复杂度 | O(n) | O(n) |

| 优缺点 | 不会消耗额为内存，但是会阻塞客户端 | 不阻塞客户端，需要fork 消耗内存 |

  1. 自动生成RDB 在某些条件达到的时候，自动生成 (内部生成bgsave)

| 配置 | seconds(在指定时间/秒) | changes (改变数据条数) |

| —- | ———————- | ———————- |

| save | 900 | 1 |

| save | 300 | 10 |

| save | 60 | 10000 |

Rdb 文件配置

dbfilename: dump.rdb(生成的rdb文件名称)

dir ./(生成文件的路径)

stop-writes-on-bgsave-erro yes (在bgsave 保存的时候是否停止写入)

rdbcompression yes (是否已压缩的格式写入rdb文件)

rdbchecksum yes (是否rdb文件进行检验)

最佳配置

  1. 不使用默认自动生成rdb文件
  1. dbfilename ${port}.rdb 以端口号命名
  1. dir /bigdiskpath 特定的目录
  1. stop-writes-on-bgsave-erro yes bgsave 发生错误的时候停止写入

​

RDB 总结

  1. RDB 是redis 内存到硬盘的快照，用于持久化

  2. save 通常会阻塞Redis

  3. basave 不会阻塞Redis 但是会fork 新进程

  4. save 自动配置满足任一就会被执行

  5. 比较耗时耗性能

  6. 容易丢失数据

Redis 的主从复制

设置主从关系

  1. 临时主从关系

    SLAVEOF 主ip 主端口

  2. 永久主从

    redis. config 中添加 SLAVEOF 主ip 主端口

  3. 从库只读不写

    slaveof-read-only：yes/on

主从复制原理

  1. 当主redis和从redis 建立MS 关系的时候，从库会向主发送PSYNC 命令
  2. 主库收到PSYNC 命令后会开始在后台保存快照数据(rdb 快照)并且在快照期间写的命令加入缓存，在快照写入完毕的时候将快照和缓存一并发送给从redis
  3. 从redis 接收到快照和缓存后 ，载入快照文件并写入缓存命令
  4. 之后每次主redis 进行写操作事，都会将命令发送给从redis 从而保证数据的一致性

当redis 宕机后处理

  1. 当从redis 宕机时 ，处理方式
    • 从redis 重新启动后会自动加入主从架构中，自动完成同步数据
    • redis 2.8 就实现类 主从断线后恢复为增量复制
  2. 主redis 宕机
    1. 当主redis 宕机后，先在从redis 中执行 SLAVEOF NO ONE 命令，断开主从关系并提升为主库，然后重启宕机的主库 并执行SLAVEOF 命令 降为从库,这样数据就能同步起来

Redis 的哨兵机制

  • 哨兵的作用就是对redis 集群的监控，它是一个独立的进程

  • 哨兵的功能

    1. 监控主从数据库是否正常
    2. 主数据出现的故障后自动将从数据库转为主数据库

  • 原理

  • 启动哨兵命令

    Redis -sentinel sentinel.config
